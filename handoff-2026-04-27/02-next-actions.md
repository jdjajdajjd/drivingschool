# Что делать дальше

## Самое первое

1. Проверить рабочее дерево:

```bash
git status --short
```

2. Запустить проект:

```bash
npm install
npm run dev
```

3. Открыть:

```text
http://localhost:5173/school/virazh
```

4. Прогнать проверки:

```bash
npm run typecheck
npm run build
```

## Приоритет 1: оставить один Cloudflare project

Нужно убрать путаницу с двумя проектами.

Оставить:

```text
drivingschool
https://drivingschool-6wy.pages.dev
```

Удалить после подтверждения пользователя:

```text
drivedesk
https://drivedesk.pages.dev
```

Важно: удаление Cloudflare project - действие с потерей облачного проекта, поэтому без явного подтверждения не удалять.

После удаления:

- проверить, что пуш в `main` деплоит только `drivingschool`;
- проверить, что GitHub Actions только собирает проект;
- обновить README/доки, если где-то остался `drivedesk.pages.dev`.

## Приоритет 2: довести Supabase до нормальной основы

Сейчас Supabase подключен, но архитектура переходная. Надо двигаться к тому, чтобы Supabase был единственным источником правды.

Сделать:

- публичную страницу школы читать напрямую из Supabase;
- админку читать напрямую из Supabase или через нормальный service layer;
- убрать зависимость важных сценариев от localStorage;
- оставить localStorage только для локального профиля/сессии ученика, если пока нет auth;
- добавить нормальные функции для создания/редактирования:
  - филиалов;
  - инструкторов;
  - слотов;
  - учеников;
  - настроек школы.

SQL держать в одном файле:

```text
supabase/DRIVEDESK_FULL_SETUP.sql
```

Пользователь просил не раскидывать SQL по разным файлам.

## Приоритет 3: авторизация и роли

Без авторизации продукт нельзя продавать.

Минимальные роли:

- superadmin DriveDesk;
- директор автошколы;
- админ автошколы;
- инструктор;
- ученик.

Минимальные сценарии:

- директор входит в админку;
- ученик входит/создает профиль по телефону;
- инструктор видит свои занятия;
- чужие школы не видят данные друг друга.

Supabase Auth можно использовать, но нужно аккуратно продумать UX:

- для ученика лучше вход по телефону/коду;
- для директора можно email/password на первом этапе;
- для MVP можно начать с magic link/email, но для автошкол удобнее телефон.

## Приоритет 4: нормальный профиль ученика

Нельзя показывать ученику фейковые данные.

Профиль должен показывать только то, что реально известно:

- имя;
- телефон;
- ближайшая запись;
- список будущих записей;
- список прошедших занятий;
- кнопка записаться;
- расписание автошколы;
- закрепленный инструктор, если он реально задан;
- закрепленный филиал, если он реально задан;
- остаток часов, если автошкола реально ведет часы.

Пока не подключена логика часов, не показывать "остаток часов".

Пока нет настоящих навыков, не показывать "прогресс навыков".

Пока нет теории/материалов, не показывать эти блоки.

## Приоритет 5: расписание автошколы

Пользователь хочет подробный календарь на 2 недели вперед.

Нужно сделать:

- отдельный экран/блок расписания;
- вид по дням;
- слоты свободные/занятые;
- фильтры:
  - филиал;
  - инструктор;
  - коробка/машина;
  - только свободные;
- понятные статусы;
- без перегруза.

Важно: для ученика расписание должно быть просто понятным. Для админа - более плотным и рабочим.

## Приоритет 6: booking groups для нескольких занятий

Сейчас есть поддержка выбора нескольких слотов, но подтверждение и админка еще не идеально отражают группу.

Нужно:

- если ученик выбрал 2 занятия, создавать одну группу записи;
- на подтверждении показывать все выбранные занятия;
- в профиле показывать их как список;
- в админке видеть, что они созданы вместе;
- отменять/переносить по одному занятию, но понимать группу;
- если один слот уже занят в момент записи, ничего не создавать и показать нормальную ошибку.

## Приоритет 7: админка директора

Для продажи автошколе директор должен уметь:

- настроить школу;
- настроить цвет/логотип;
- создать филиалы;
- создать инструкторов;
- создать расписание;
- массово создать слоты;
- задать длительность занятия;
- задать лимит занятий за одну запись;
- задать, выбирает ли ученик филиал сам;
- закрепить ученика за филиалом;
- закрепить ученика за инструктором;
- видеть записи;
- отменять/переносить/завершать записи;
- видеть учеников;
- видеть историю ученика.

Все эти действия должны сохраняться в Supabase.

## Приоритет 8: безопасность базы

Сейчас RLS и политики пока пригодны для разработки, но не для настоящего запуска.

Нужно:

- закрыть публичное чтение лишних таблиц;
- запретить публичное чтение всех bookings;
- сделать политики по school_id;
- связать пользователей Supabase Auth с ролями;
- сделать таблицу memberships/roles;
- проверить, что ученик видит только свои данные;
- проверить, что директор видит только свою школу.

## Приоритет 9: визуальная полировка

Пользователь недоволен "обычным" видом. Нужно сделать красивее, но не шумно.

Принципы:

- спокойный синий брендовый акцент;
- крупные понятные кнопки;
- чистые карточки;
- без декоративного мусора;
- без фейковых метрик;
- текст короткий;
- один главный CTA на экран;
- на мобильном все должно читаться без напряжения.

Не надо:

- кислотные цвета;
- много иконок без смысла;
- много карточек ради карточек;
- лендинг вместо полезного сценария;
- большие маркетинговые блоки на странице записи.

## Приоритет 10: QA перед продажей

Проверить сценарии:

- новый ученик записался без профиля;
- новый ученик записался и создал профиль;
- повторный вход с профилем;
- запись на один слот;
- запись на два слота в один день;
- запись на два слота в разные дни;
- попытка записаться на уже занятый слот;
- админ отменяет запись;
- админ завершает запись;
- админ переносит запись;
- календарь скачивается/добавляется;
- мобильный экран 375px;
- desktop;
- обновление страницы после записи;
- другой браузер/инкогнито.

Проверки:

```bash
npm run typecheck
npm run build
```

## Что не забыть

- Не коммитить `.env.local`.
- Не записывать service role key в документацию.
- Не вставлять реальные ключи в markdown.
- SQL-изменения держать единым файлом.
- Перед удалением Cloudflare project спросить явное подтверждение.
- После каждого важного изменения проверять реальный prod URL.
