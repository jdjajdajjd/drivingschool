# Полный контекст DriveDesk

## Что это за продукт

DriveDesk - SaaS для автошкол.

Идея продукта: автошкола получает простую страницу записи, админку, расписание, управление учениками, инструкторами и занятиями. Ученик может записаться на практическое занятие без звонков и лишних шагов.

Ключевой фокус пользователя: интерфейс должен быть понятен людям 40+, без визуального шума, без лишних кнопок, без сложных терминов. При этом он не должен выглядеть сыро и скучно.

Референс по ощущению: приложение "Хочу водить". Не копировать буквально, но взять принципы:

- понятный личный кабинет ученика;
- крупные карточки;
- спокойная мобильная структура;
- расписание как главный сценарий;
- минимум лишней информации;
- полезные данные вместо декоративных блоков.

## Что сделали по UI записи

Изначальная задача была сделать запись простой для людей 40+.

Сделано:

- упрощен публичный navbar;
- убраны лишние ссылки и кнопки на публичной странице;
- страница `/school/:slug` стала основным местом для ученика;
- запись больше не должна быть "в лоб" сразу при входе, если у ученика уже есть профиль;
- есть кнопка `Записаться на занятие`, которая запускает сценарий записи;
- филиал выбирается кликом и сразу ведет дальше;
- инструктор выбирается кликом и сразу ведет дальше;
- можно выбирать несколько дат, если это разрешено настройками школы;
- можно выбирать несколько слотов, если директор поставил лимит больше одного;
- длительность показывается человечески: `1 час 30 минут`, а не `90 минут`;
- кнопка календаря называется `Добавить в календарь`, а не `Скачать в календарь`;
- финальная кнопка называется по смыслу: `В профиль ученика` или `К записи на занятие`;
- после записи у пользователя спрашивают, создавать ли профиль ученика по введенным имени и телефону.

## Важная продуктовая логика по профилю ученика

Пользователь явно сказал:

- не надо сразу показывать новому человеку фейковый профиль;
- профиль должен появляться только после записи и согласия создать профиль;
- перед экраном "вы записаны" надо спросить: `Создать профиль ученика?`;
- если человек создал профиль, при следующем входе на страницу школы он видит личный кабинет;
- если профиль не создан, человек не должен видеть лишний кабинет.

Из профиля надо убрать фейковые и бессмысленные данные:

- прогресс навыков без реального источника;
- теорию и материалы пока убрать;
- "инструкторы" как отдельный счетчик не нужен, потому что ученик обычно ходит к одному инструктору;
- остаток часов показывать только когда это реально известно из базы/админки;
- не выдумывать данные, которых автошкола не дала.

Что в профиле должно быть полезным:

- ближайшая запись;
- кнопка записаться еще;
- расписание автошколы;
- данные ученика;
- в будущем: остаток оплаченных часов, если админка это ведет;
- в будущем: закрепленный филиал/инструктор, если автошкола это настроила.

## Что сделали по Supabase

Подключили Supabase как базу.

Добавлено:

- `@supabase/supabase-js`;
- `src/lib/supabase.ts`;
- `src/lib/supabaseTypes.ts`;
- `src/services/supabasePublicService.ts`;
- `src/services/supabaseAdminService.ts`;
- `src/services/supabaseSync.ts`;
- `supabase/DRIVEDESK_FULL_SETUP.sql`;
- `scripts/apply-supabase-sql.mjs`;
- `.env.example` с шаблонами переменных.

Единый SQL-файл:

```text
supabase/DRIVEDESK_FULL_SETUP.sql
```

Пользователь просил все SQL-изменения держать одним файлом, чтобы можно было целиком заменить/выполнить в Supabase SQL Editor, а не собирать по кускам.

Сейчас SQL создает:

- schools;
- branches;
- instructors;
- students;
- slots;
- booking_groups;
- bookings;
- slot_locks;
- демо-школу `Автошкола «Вираж»`;
- филиалы;
- инструкторов;
- учеников;
- слоты на 14 дней;
- функции для создания, отмены, завершения и переноса записи.

Сделанные Supabase-функции:

- `public_create_booking`;
- `public_cancel_booking`;
- `public_complete_booking`;
- `public_reschedule_booking`.

Публичная запись теперь создает booking в Supabase.

Админские действия отмены/завершения/переноса записи теперь тоже отправляются в Supabase.

Приложение при загрузке подтягивает данные из Supabase и синхронизирует их в старую локальную модель, чтобы текущие страницы админки продолжали работать.

## Важное ограничение текущей архитектуры

Сейчас проект в переходном состоянии.

Supabase уже подключен, но не все страницы полностью переписаны на прямую работу с Supabase.

Есть промежуточный слой:

- Supabase = источник данных;
- данные подтягиваются в localStorage-модель;
- старые страницы админки читают из этой модели.

Это нормальный временный мост, но не финальная архитектура для продажи.

Что еще не идеально:

- нет полноценной авторизации;
- нет ролей директор/админ/инструктор/ученик;
- RLS-политики Supabase пока dev-уровня, не production;
- часть админских create/update операций еще localStorage-first;
- слот-локи в UI еще не полноценные server-side locks;
- multi-slot confirmation показывает в основном первую запись, а не красивую группу занятий;
- нет настоящего SMS/Telegram/email подтверждения;
- нет оплаты;
- нет нормального tenant onboarding для разных автошкол.

## Cloudflare и почему был деплой в два места

Было два Cloudflare Pages проекта:

1. `drivingschool`
   - URL: `https://drivingschool-6wy.pages.dev`
   - подключен к GitHub напрямую;
   - сам деплоится при пуше в `main`;
   - сейчас это рабочий проект.

2. `drivedesk`
   - URL: `https://drivedesk.pages.dev`
   - без Git connection;
   - туда деплоил GitHub Actions workflow;
   - из-за этого один пуш выглядел как два деплоя;
   - проект уже удалён.

Что уже сделали:

- `.github/workflows/deploy.yml` больше не деплоит в `drivedesk`;
- workflow переименован в `Build` и только проверяет `npm run build`;
- деплой теперь должен идти только через Cloudflare Git connection проекта `drivingschool`.

Что осталось:

- пользоваться только `drivingschool-6wy.pages.dev`;
- в будущем подключить красивый домен.

## Что пользователь хочет дальше
