# Launch plan

## Цель

Довести DriveDesk до состояния production pilot: без видимых заглушек, без demo-localStorage как источника правды, с понятным ученическим профилем, надежной записью, админскими настройками и одним стабильным деплоем.

## 1. Backend and data

Сейчас главный blocker запуска - `localStorage`. Его нужно заменить на реальное хранилище.

Рекомендуемый вариант для быстрого запуска:

- Cloudflare Pages для фронта;
- Cloudflare Pages Functions или Workers для API;
- Cloudflare D1 для данных.

Минимальные сущности:

- `schools`
- `branches`
- `instructors`
- `students`
- `slots`
- `bookings`
- `booking_groups`
- `school_booking_policies`
- `admin_users`
- `student_sessions`

Правило: публичный UI, профиль, админка и подтверждение читают одни и те же данные из backend API.

## 2. Auth

Нужна настоящая авторизация.

Минимум для pilot:

- ученик входит по телефону;
- подтверждение кодом;
- после входа видит свой профиль;
- админ входит отдельно;
- роли: `student`, `school_admin`, `superadmin`.

Если SMS не подключаем завтра, нельзя называть это production. Тогда нужно явно считать это закрытым pilot/demo и не обещать отсутствие заглушек.

## 3. Student profile

Профиль должен стать главным экраном ученика.

Содержимое:

- ближайшая запись;
- активные записи;
- история занятий;
- остаток часов, если эти данные есть;
- закрепленный филиал;
- инструктор;
- кнопка `Записаться на занятие`;
- кнопка изменения данных;
- понятные пустые состояния.

Не показывать фейковый прогресс и фейковые часы.

## 4. Booking flow

Сценарий записи:

1. Ученик нажимает `Записаться на занятие`.
2. Если филиал закреплен, шаг филиала пропускается.
3. Если филиал не закреплен, клик по филиалу сразу ведет дальше.
4. Клик по инструктору сразу ведет к датам.
5. Даты и слоты можно выбирать в рамках лимита директора.
6. На последнем шаге показывается краткое резюме.
7. Создается `booking_group` и все `bookings` внутри него.
8. Подтверждение показывает все занятия из группы.

Критично: создание нескольких занятий должно быть атомарным. Нельзя создать одно занятие из двух и показать успех.

## 5. Admin settings

Добавить полноценную настройку правил:

- выбор филиала учеником: да/нет;
- закрепленный филиал ученика;
- максимальное число занятий за одну запись;
- можно ли выбирать несколько дней;
- длительность занятия;
- окно записи вперед;
- минимальное время до отмены;
- минимальное время до переноса;
- показывать ли инструкторов;
- разрешать ли выбирать инструктора.

Карточка ученика:

- филиал;
- инструктор;
- остаток часов;
- активные записи;
- история;
- заметки.

## 6. Confirmation

Подтверждение должно работать для одной и нескольких записей.

UI:

- заголовок: `Запись сохранена`;
- список занятий;
- данные ученика;
- `Добавить в календарь`;
- `В профиль ученика`.

ICS:

- одно занятие = один event;
- несколько занятий = несколько events в одном файле.

## 7. Visual polish

Дизайн должен быть спокойным, но не скучным.

Направление:

- мобильный first;
- светлый фон;
- один синий акцент;
- аккуратные карточки;
- крупная типографика;
- минимум вторичного текста;
- только функциональные иконки;
- никаких случайных цветов и декоративного мусора.

Особое внимание:

- профиль ученика;
- выбор даты/слотов;
- подтверждение;
- пустые состояния;
- ошибки.

## 8. Deployment

Оставить один Cloudflare Pages project.

Рекомендация:

- оставить `drivingschool` как Git-connected project;
- удалить/отключить GitHub Actions деплой в `drivedesk`;
- или наоборот, но не оба сразу.

После исправления:

- push в `main` = один production deploy;
- PR = preview deploy;
- production URL понятный и стабильный.

## 9. Quality gate

Перед запуском:

```bash
npm run typecheck
npm run build
```

Ручной QA:

- первый вход ученика;
- повторный вход;
- запись на 1 занятие;
- запись на 2 занятия в один день;
- запись на 2 занятия в разные дни;
- закрепленный филиал;
- свободный выбор филиала;
- нет слотов;
- слот заняли во время записи;
- подтверждение;
- календарь;
- отмена;
- перенос;
- админские настройки.

## Definition of done

Можно считать готовым к pilot launch, когда:

- нет demo/localStorage как источника правды;
- нет мертвых кнопок;
- нет фейковых данных;
- каждый empty/error state понятный;
- запись не ломается при конфликте слотов;
- деплой один;
- весь основной flow проходит на телефоне без подсказок разработчика.
